📖 ちなみに解説

OpenClaw / NanoClaw / Hermes Agent
3 つのエージェント・フレームワークの違い

このハンズオンでは Hermes Agent を採用しました。同じ「ローカルで動く AI エージェント」枠の選択肢として OpenClaw・NanoClaw もあります。それぞれの設計思想と、今回 Hermes を選んだ理由をまとめます。

🧩 3 つの全体像

どれも「LLM をオーケストレーションしてメッセージングプラットフォームと繋ぐ」役割は同じ。違いは設計の力点。

🏛
OpenClaw
標準的・多機能・歴史長め

長くから存在する AI エージェントフレームワーク。設定ファイル (JSON / YAML) ベースで、多くのチャネル・LLM プロバイダに対応。標準的だが、機能数の多さゆえセットアップは重め。

強み
  • 歴史長く、ナレッジが豊富
  • 多機能 (プラグイン・スキル多数)
  • 多くの企業導入実績
弱み
  • 設定項目が多く初学者が迷う
  • 設定の複雑さゆえ攻撃面 (config injection 等) が広いという指摘もある
  • LINE 公式対応の状況は限定的
🔬
NanoClaw
極小・コンテナ隔離・個人所有

OpenClaw の「監査できない複雑さ」への反動として作られた軽量フレームワーク。~15 ファイルの極小実装、Docker コンテナで OS レベルの隔離を提供。Claude Agent SDK ベース。

強み
  • コード規模が小さく全部読める
  • Docker サンドボックスで安全
  • 個人所有・カスタマイズ前提の設計
弱み
  • LINE 非対応 (skill 自作で 2〜3 日)
  • Docker 必須でセットアップが重い
  • 設定ファイルなし思想 (Claude Code に都度コード書き換えさせる)
🚀
Hermes Agent
自己改善・多プラットフォーム

Nous Research 開発。"育つ AI" のコンセプトで永続メモリを持ち、22 メッセージングプラットフォームに公式対応。LLM プロバイダも豊富で Ollama を含む。YAML 設定で扱いやすい。

強み
  • LINE が v0.14 以降公式同梱
  • Docker 不要 (curl 一発インストール)
  • Ollama / OpenAI / Anthropic 等 22+ LLM
  • 22 メッセージング platform 対応
弱み
  • 機能多すぎて全貌把握は大変
  • tool-calling 前提なので小さいモデルでは指示追従に弱い
  • Windows ネイティブは早期ベータ (WSL2 推奨)

📊 なぜ今回は Hermes を選んだか

「シンプル & LINE 対応」が最大の決め手。受講者のセットアップ負荷を最小化したかった。

⚡ セットアップの簡単さ
curl 一発でインストール → YAML 一行で Ollama 連携。 NanoClaw は Docker Desktop + Sandboxes 機能の有効化が必要で、教室の Wi-Fi で詰みやすい。 Hermes 採用
💬 LINE 公式対応
Hermes は plugins/platforms/line/ に LINE アダプタを同梱 (v0.14.0 以降)。 NanoClaw は LINE 非対応で、対応するには Channel adapter を自作 (実装 2〜3 日)。 授業の本筋は「設定の簡便さ」なので、NanoClaw を選ぶと脇道に逸れる。 Hermes 採用
🪶 Docker 不要
Hermes は local バックエンドがデフォルトで Docker 不要。 NanoClaw は Docker 必須でディスク数 GB と Sandboxes 機能の前提あり。 低スペック PC でも始められる柔軟性が要件だったので Hermes が有利。 Hermes 採用
🧠 ローカル LLM 連携
Hermes は OpenAI 互換エンドポイントを base_url で指定するだけで Ollama を見られる。 OpenClaw は設定が散らかっているし、NanoClaw はそもそも LLM 接続も Claude Agent SDK 経由で固有。 Hermes 採用
📖 学習素材として
コードを全部読みたいなら NanoClaw が最強 (~15 ファイル)。OpenClaw は機能網羅向き。Hermes は中間。 初回ハンズオンとしては「動くまでが速い」が最優先と判断して Hermes。 慣れた後、NanoClaw でコード読解に挑戦するのは強くおすすめ。 Hermes 採用 (NanoClaw は次の学び)

🔄 後で乗り換える時のヒント

ハンズオン完了後、別の系統を試したくなったら以下を参考に。

Hermes → NanoClaw に乗り換えたい

共通の概念 (LLM + チャネル + Webhook + tunnel) は同じなので、知識は移植できる。

  • NanoClaw は Claude Agent SDK ベースなので、Anthropic API キーが必要 (Ollama も ANTHROPIC_BASE_URL 経由で接続可)
  • Docker Desktop 4.58+ と "Docker Sandboxes" 機能を有効化
  • LINE 用の skill が無いので、Telegram / Discord などへチャネル切替が現実的
  • Claude Code に「NanoClaw に乗り換えたい、現在の Hermes 設定を踏まえて手順を提案して」と頼むのが最短
Hermes → OpenClaw に乗り換えたい

多機能なので「Hermes でできなかった〇〇がやりたい」という具体的動機がある時に試すのが良い。漠然と乗り換えると複雑さに溺れる。

  • JSON 設定ファイル中心 (Hermes の YAML より項目が多い)
  • プラグインシステムが充実 (Discord/Slack/Email など)
  • セキュリティ的に古い構造の指摘もあるので、本番運用前にレビュー推奨
全部やめて「自作 Webhook receiver」で書きたい

勉強として最も学べる。Express/FastAPI で 100 行程度の webhook receiver を書き、Ollama を直接叩く。

  • 署名検証 (HMAC-SHA256) を自分で実装
  • LINE Reply API への HTTPS POST も自分で書く
  • フレームワークが何を抽象化してくれていたかが体感で分かる
  • Claude Code に「Hermes を使わずに LINE bot を素の Node.js で書き直して」と頼むと壁打ち相手になる
📝 メモ — フレームワーク選定は「文脈次第」

このページの結論「Hermes が良い」は、本ハンズオンの目的 (60 分で LINE 連携のローカル翻訳 Bot を完成させる) に対しての話。
用途が変われば最適解も変わります。例えば「企業内で Slack のみ運用、セキュリティ重視」なら NanoClaw + Docker が最適だし、「複雑なワークフロー自動化」なら OpenClaw が当てはまる場面もある。
「最強のフレームワーク」は存在しない。文脈で選ぶのが正しい。
