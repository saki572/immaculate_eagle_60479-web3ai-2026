Step 2 でやること（概要）

ローカルで動くアプリを「世界からアクセスできるアプリ」にします。
Vercel は GitHub と連携するだけで自動的に Next.js を本番ビルド → デプロイ → 公開URL発行してくれるサービス。
受講者は GitHub と Vercel のアカウントさえあれば、コマンドラインを触らずに公開できます。

🎯 このステップでやること

  1. GitHub アカウント準備

    無料でOK。受講者は自分の本名 or ニックネームで取得済みであることが多い

  2. サンプルレポを fork or 自分のレポに push

    テンプレートから自分のレポを作る or ローカルで clone したやつを自分のレポに push し直す

  3. Vercel アカウント準備

    GitHub アカウントで sign in（数十秒）

  4. Vercel に Import

    「New Project」→ 自分のレポを選ぶ → Next.js が自動検出される

  5. 環境変数を入力

    GEMINI_API_KEY を Vercel の Environment Variables に登録

  6. Deploy ボタン

    1〜2 分待つと公開URL（例: https://your-app.vercel.app）が出る

  7. 動作確認

    公開URLで質問を投げて minta先生から回答が返ってくれば完了

# Step 2: Vercel にデプロイ

Step 1 でローカルで動いた「教えて、minta先生」アプリを、Vercel にデプロイして公開URLで動くようにします。

## 私の状況
- Step 1 完了: ローカル http://localhost:3000 で動いている
- GitHub アカウント: __ (例: github.com/your-username)
- Vercel アカウント: __ (まだ無ければ取得から)

## ゴール
- 公開URL (例: https://minta-sensei-handson.vercel.app) で minta先生が答える
- Vercel ダッシュボードで環境変数を管理できる状態

## 進め方

### 1. GitHub に自分のレポを作る
- まず GitHub にログインして、新規レポ (例: minta-sensei-handson) を Public で作成
- ローカルの clone 元 (web3ai-gairon/minta-sensei-handson) の remote を、自分のレポに付け替え:
  - `git remote set-url origin git@github.com:YOUR-USERNAME/minta-sensei-handson.git`
  - もしくは `git remote add mine git@github.com:YOUR-USERNAME/minta-sensei-handson.git`
- `git push -u origin main` (or `git push -u mine main`) で push

### 2. .gitignore 確認
- `.env.local` が .gitignore に含まれていることを再確認
- 万が一すでにコミットしていないか `git log --all -- .env.local` で確認
- 心配なら Claude Code に「.env.local が GitHub に push されていないか確認して」と頼む

### 3. Vercel に Import
- vercel.com → Sign in with GitHub
- New Project → 作ったレポを選ぶ → Import
- Framework Preset: Next.js が自動検出されるはず
- Root Directory: そのまま (./)

### 4. 環境変数を入力
- 「Environment Variables」セクションを開く
- Name: GEMINI_API_KEY
- Value: 自分の Gemini API キー (sk-... ではなく AIza... で始まる Google のキー)
- **重要: キーを私 (Claude Code) のチャットに貼らせない。Vercel ダッシュボードに私が直接入力する**
- 全環境 (Production / Preview / Development) にチェック

### 5. Deploy
- Deploy ボタンを押す
- 1-2 分待つ
- 「Visit」ボタンが出たら、公開URLが完成

### 6. 動作確認
- 公開URLで質問を投げて、minta先生が答えるか確認
- ダメだったら Vercel の Deployments タブ → 失敗したデプロイをクリック → Runtime Logs をチェック
- ログを私が読んで、原因と対処を伝える

## セキュリティ注意
- API キーをチャットに貼らせない (Vercel ダッシュボードに直接入力させる)
- Vercel の環境変数は「Sensitive」フラグを付けると、ダッシュボード上でも値が伏せられる
- 公開URLは世界からアクセス可能。本番運用前に rate limit / 認証を検討する

それでは Step 2 を始めてください。

🔐 環境変数についての解説

「環境変数って何？」が初めての人向け。

環境変数とは プログラムが OS から読み取れる「設定値」のこと。コードに直接 API キーを書く代わりに、環境変数として外から渡す
なぜ使うか ① コードに秘密の値を書くと、GitHub に push したときに世界中に漏れる
② 開発環境と本番環境で値を変えたい
③ チーム開発で個々のキーを持てる
ローカルでは .env.local ファイルに書く。Next.js が process.env.GEMINI_API_KEY として読み取る
Vercel では ダッシュボード → Settings → Environment Variables で登録。デプロイ時にコンテナの環境変数として注入される
クライアントから読める変数 NEXT_PUBLIC_ から始まる名前にすると、ブラウザ側 JS からも見える（だから機密にしてはいけない）。サーバ側だけで使う秘密キーは NEXT_PUBLIC_ を付けない

🛡 セキュリティで気をつけること

  • API キーを GitHub に絶対に push しない。過去にコミットしてしまっていたら、すぐ Revoke して新しいキーに差し替え。
  • レポを Public にする前に、コミット履歴を確認: git log --all -p | grep -i 'api_key\|secret\|sk-\|AIza' 等で検索。
  • Vercel の環境変数の「Sensitive」フラグを ON にすると、ダッシュボード上でも値が伏せられる。チームで作業する場合に有効。
  • 公開URLは世界中からアクセス可能。悪用されると課金されたり、API クォータが減ったりする。本番運用するなら必ず認証 / レート制限を入れる。
  • Preview デプロイにも環境変数が共有される。PR から派生する Preview URL も世界からアクセス可能なので、機密データを扱う場合はスコープを Production だけに絞る選択肢もある。

✅ Step 2 完了チェック

  • ☐ 自分の GitHub レポにコードが push されている
  • ☐ Vercel のダッシュボードにプロジェクトが作られている
  • ☐ Environment Variables に GEMINI_API_KEY が登録されている
  • ☐ 公開URL (vercel.app) で minta先生に質問できる
  • ☐ 質問への回答が日本語で返ってくる
  • ☐ .env.local がGitHubに含まれていない
💡 ここまでで体験したこと

「自分の PC で動くアプリ」と「世界からアクセスできるアプリ」のギャップは、実は 環境変数を Vercel に登録するだけでほぼ埋まりました。
コードは変わっていません。同じコードが、同じ API キーの値を別ルートで受け取って動くだけ。これが「設定とコードを分離する」考え方の威力です。
